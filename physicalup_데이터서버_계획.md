# 💾 PHYSICAL UP 데이터 관리 · 서버 계획

> 정기검진(구독형) = 고객 데이터 계속 누적 → 저장·서버 설계 필수
> ※ 구체 요금은 업체·시점별 상이 → 구조와 용량 추정 위주. 실제 견적은 별도.

═══════════════════════════════════════
## 1. 우리가 저장할 데이터 (종류)
═══════════════════════════════════════
| 데이터 | 내용 | 민감도 |
|--------|------|--------|
| 고객 기본정보 | 이름·연락처·자녀 생년월일 | ★개인정보 |
| 설문 응답 | 24문항 답변 (회차별) | 중 |
| 인바디 수치 | 체지방·근육·체중 등 | ★건강정보 |
| 분석 결과 | BioCode·격차·처방 (회차별) | 중 |
| 발송 보고서 | PDF·웹 (회차별) | 중 |
| 결제 기록 | 거래내역 | ★금융정보 |

★주의: 미성년자 건강정보 = 민감정보
  → 개인정보보호법 준수 필수 (동의·암호화·보관기간)

═══════════════════════════════════════
## 2. 용량 추정 (1인당 → 전체)
═══════════════════════════════════════
### 1인 1회차 용량
- 텍스트 데이터(기본정보+설문+수치+결과): 약 50KB
- 보고서 PDF: 약 1~3MB
- 보고서 웹/이미지: 약 1~2MB
- → ★1인 1회차 ≈ 약 3~5MB

### 누적 추정 (정기검진=회차 누적)
| 고객수 | 연 2회 검진 | 1년 누적 | 비고 |
|--------|------------|---------|------|
| 100명 | 200건 | 약 1GB | 시작 단계 |
| 1,000명 | 2,000건 | 약 10GB | 성장 |
| 10,000명 | 20,000건 | 약 100GB | 본격 |
★텍스트만 보면 매우 작음(수십MB). 용량 대부분=보고서 파일
→ 보고서는 생성형(필요시 재생성)으로 두면 용량 大절감

═══════════════════════════════════════
## 3. 저장 구조 설계
═══════════════════════════════════════
```
[DB] 텍스트 데이터 (가벼움, 핵심)
  고객 / 회차별 설문·수치·결과 → 관계형DB
  · 고객 ─< 검진회차 ─< 설문응답·인바디·결과
  · 회차 누적 = 성장 히스토리 (비교페이지 자동생성 기반)

[파일저장소] 보고서 PDF·이미지 (무거움)
  · 옵션A: 매번 저장 (빠름, 용량↑)
  · 옵션B: 결과데이터만 저장 → 보고서는 요청시 재생성 (용량↓) ★추천
```
★핵심: 결과 데이터(작음)만 DB에 영구보관,
  보고서 파일은 재생성 가능하게 → 용량·비용 절감

═══════════════════════════════════════
## 4. 서버 선택지 (단계별)
═══════════════════════════════════════
[1차 — 베타/소규모]
  - 클라우드 관리형 (네이버클라우드·AWS·구글클라우드)
  - 또는 노코드 백엔드(파이어베이스·수파베이스) = 빠른 시작
  - 용량 적어 최소 요금제로 충분
[2차 — 성장]
  - DB + 파일저장소(S3 등) 분리
  - 트래픽 늘면 사양 업
[3차 — 본격]
  - 오토스케일링·백업 자동화·이중화

★솔로 운영 → 관리형 클라우드 추천 (서버 직접관리 부담↓)

═══════════════════════════════════════
## 5. ★개인정보·보안 (미성년 건강정보 = 필수)
═══════════════════════════════════════
- 수집·이용 동의 (부모 동의, 미성년자)
- 민감정보(건강) 별도 동의
- 암호화 저장 (개인정보·건강정보)
- 접근 권한 관리 (대표님만 열람)
- 보관기간·파기 정책
- ★개인정보보호법·관련법 준수 (전문가 자문 권장)
- 결제정보는 PG사가 처리 (직접 저장 X = 안전)

═══════════════════════════════════════
## 6. 비용 성격 (참고, 구체 견적 별도)
═══════════════════════════════════════
- 텍스트 데이터 = 매우 저렴 (용량 작음)
- 보고서 파일 = 재생성 방식이면 저장비용↓
- 초기(고객 수백): 월 몇만원 수준대로 시작 가능 (관리형)
- 고객 늘수록 단계적 증가 → 매출과 함께 감당
- ★결론: 데이터 자체는 가벼워서 초기 부담 적음
  진짜 비용은 트래픽·자동화 도구(알림톡·결제 수수료 등)

═══════════════════════════════════════
## ⏭️ 확정·자문 필요
═══════════════════════════════════════
1. 클라우드 업체 선택 (네이버클라우드 국내·간편 / AWS 범용)
2. 개인정보 처리방침 작성 (전문가 자문)
3. 보고서: 매번저장 vs 재생성 (용량 정책)
4. 백업·보안 수준
5. ★실제 견적은 업체 상담 (용량·트래픽 산정 후)

## 7. ★튕김 방지 (대박 나도 안 터지게) — 솔로 사업자용
═══════════════════════════════════════
> 대표님 걱정: "대박 났는데 튕기면? 1인 사업이라 손쓸 방법 없음"
> → 손 안 써도 알아서 버티는 구조로 설계 (자동 확장)

### 핵심: 관리형 = 대표님 손 안 댐
- ★오토스케일링(auto-scaling): 손님 몰리면 서버 자동 증설, 빠지면 감소
  · 대표님 아무것도 안 함 → 클라우드가 알아서
  · 옛날식(서버 1대 직접관리)과 정반대
- 서버리스(Serverless): 서버 자체를 신경 안 씀
  · 쓴 만큼만 비용, 안 쓰면 0 → 대박 전엔 거의 공짜

### 안 터지는 안전장치 3개
```
1. 큐(Queue) 줄세우기
   동시 1000명 신청 → 순서대로 처리 → 시스템 안 터짐
   (대표님 검수가 밀릴 뿐, 다운 안 됨)

2. 비동기 처리 "나중에 보내드려요"
   신청 즉시 결과 X → "분석 후 카톡 발송"
   → 실시간 부하 없음 ★우리 사업과 딱 맞음

3. 정적 보고서(링크)
   웹페이지라 수천 명 열람해도 가벼움
```

### ★우리 사업은 원래 튕김 위험 낮음
- 실시간 게임서버 아님 (1초 안에 다 처리할 필요 X)
- 신청 → 검수 → 카톡발송 = '시차'가 자연스러움
- 부모도 "분석은 시간 걸리지" 하고 기다림
- → 줄 세워 차근차근 처리해도 문제없음

### 대박 시나리오 대응
| 상황 | 자동 대응 | 대표님 할 일 |
|------|----------|------------|
| 평소 (수십명) | 최소 사양 | 검수만 |
| 갑자기 몰림 (수백) | 자동 증설 | 검수만 (밀리면 순차) |
| 대박 (수천) | 자동 대폭 증설+큐 | 검수 도우미 고용 고려 |
★시스템은 자동, 대표님은 '검수 속도'만 신경
  (검수 밀리면 → 알바/가족 검수 추가 = 사람 문제지 서버 문제 아님)

### 실무 권장
- 관리형 클라우드/서버리스로 시작 (네이버클라우드·파이어베이스·버셀 등)
- 트래픽 폭증 자동 대응 옵션 켜두기
- ★대표님은 서버 걱정 X → 콘텐츠·검수에 집중
- 진짜 대박나면: 그때 개발자 외주/채용 (행복한 고민)

## 8. ★★ 컷오프 자가보정 구조 (대표님 핵심 통찰) — 2026-06-24
═══════════════════════════════════════
> 대표님: "무수히 많은 아이가 하면 현실 나이대별 평균에 수렴해야 함.
>          그 범위를 1·2·3 컷오프로. 백분위는 데이터 쌓이며 보정.
>          무료 테스트 데이터가 쌓이면 보정되는 것."

### 핵심: 무료 테스트 = 데이터 수집 = 엔진 자가학습
```
무료 BioCode 테스트 (콘텐츠 유입)
   ↓ 수천 명 점수 분포 수집
"우리 저장력 하위 5% 지점 = ?점" 파악
   ↓
컷오프 자동 보정 (현실 분포 5%/85%/95%에 맞춤)
   ↓
엔진이 현실에 수렴 → 정확도 ↑
   ↓
유료 보고서 = 보정된 엔진 → 신뢰도 ↑
```

### 2단계 컷오프 전환
- [초기] 절대점수 컷오프 (40/65 임시) — 데이터 0일 때
- [전환] 백분위 컷오프 — 데이터 쌓이면
  · 우리 점수 분포의 5%·85% 지점 점수 = 컷오프
  · 현실 통계(저체중5%/정상5~85%/과체중·비만85%+)에 정렬
- ★흡수·활동: 현실 통계 애매 → 베타로 정규분포화
- ★저장: BMI 통계 있음 → 5%/85% 백분위 바로 적용 가능

### ★사업적 의미 = 데이터 해자(moat)
- 무료 테스트 = 미끼가 아니라 '엔진 학습 데이터 수집기'
- 공짜로 풀수록 → 데이터 쌓임 → 엔진 정확 → 경쟁력 ↑
- 시간이 갈수록 강해짐:
  · 초기: 임시 컷오프, 그럭저럭
  · 1년: 수천명 → 백분위 보정 → 정확
  · 3년: 수만명 → 나이·성별·종목별 정밀분포 → 압도적
- ★후발주자 추격 불가 (데이터가 해자)
- 무료 테스트 많이 풀수록 격차 ↑

### 데이터 수집 설계 (무료 테스트)
- 무료 테스트도 입력값 저장 (익명 가능, 개인정보 최소)
- 저장 항목: 나이·성별·종목 + 4입력 점수 + 최종 BioCode
- 주기적 분포 분석 → 컷오프 재계산 → 엔진 업데이트
- ★단, 미성년 정보 = 부모 동의·익명화 (개인정보보호)

## 9. ★★ 무료-유료 정합성 (대표님 핵심 통찰) — 2026-06-24
═══════════════════════════════════════
> 대표님: "무료 테스트 설문도 문제가 같아야 하고, 거기서 추론하는
>          BioCode도 반 이상은 유료랑 맞아야 됨. 무료도 생년월일·설문은
>          하니까. 인바디만 빠지는 것."

### 핵심: 무료와 유료는 같은 뿌리
```
무료 = 기질(생년월일) + 설문            → BioCode 추론 (3입력)
유료 = 기질(생년월일) + 설문 + 인바디    → BioCode 정밀 (4입력)
차이 = 인바디(비중 25%) 하나뿐
```

### 필수 원칙
1. ★무료 설문 문항 = 유료 설문 문항 (동일)
   - 같은 자로 재야 무료↔유료 비교 가능
2. ★무료 BioCode ≈ 유료 BioCode (반 이상 일치)
   - 입력 75%(설문·기질·유전)가 같으니 당연히 비슷해야 정상
   - 완전 딴판이면 = 엔진 오류 신호

### 인바디 비중 검증 도구로 활용
- 인바디 비중(현재 25%)이 적정한지 무료/유료 비교로 검증:
  · 무료↔유료 너무 다름 → 인바디 비중 과다 → 낮춤
  · 무료↔유료 거의 같음 → 인바디 역할 약함 → 유료 가치 부족
  · ★황금점 = "적당히 다르되 반 이상 일치" (대표님 기준)

### 무료→유료 전환 심리
- 무료: "BioCode 232! 인바디 넣으면 더 정확해져요"
- 부모: "232? 인바디 넣으면 뭐가 달라지지?" → 결제
- ★무료 = 유료의 신뢰 가는 맛보기 (75% 동일 → 신뢰 / 25% 차이 → 궁금증)

### 데이터 일관성
- 무료·유료 같은 설문·기질 입력 → 같은 DB 누적
- 무료 데이터도 컷오프 보정에 사용 가능 (인바디 제외 분포)
- 인바디 있는 유료 = 별도 분포로 교차검증

### ★무료 결과지 핵심 카피 (대표님 확정)
```
"이 결과는 75% 정확도입니다.
 더 정확한 분석은 유료 상세 상담으로 만나보세요."
```
- 75% 근거 = 설문55+기질12+유전8 (무료로 잡는 비중)
- 25% = 인바디(측정 필요) = 유료 차별점
- ★정직 마케팅: 거짓 없이 75% 공개 → 신뢰 ↑ → 나머지 25% 궁금증 → 전환
- 그냥 숫자 아님 = 엔진 비중에서 나온 진짜 수치 (근거 있음)
- 무료→유료 전환의 핵심 문구

## 10. ★무료/유료 설문 — 겉은 달라도 속은 같게 (대표님) 2026-06-24
═══════════════════════════════════════
> 대표님: "설문 문항도 조금은 어린이틱하게 바꿔서 줄 순 있지만,
>          추구하는 방향은 같아야지."

### 원칙: 옷(표현)은 바꿔도, 속(측정대상)은 동일
```
[속 - 반드시 같음]
  측정 대상 = 흡수/활동/저장, 몸/습관/정신
  → 다르면 무료↔유료 비교·데이터통합 불가

[겉 - 바꿔도 됨]
  문항 표현 = 무료는 가볍고 어린이틱하게
  → 같은 걸 묻되 말투만
```

### 예시 (같은 흡수-식욕)
- 유료: "한 번에 먹는 양이 또래보다 많은 편이다"
- 무료: "밥 먹을 때 '한 그릇 더!' 자주 외친다 🍚"
- → 둘 다 식욕(흡수) 측정, 표현만 친근

### 이렇게 하는 이유
1. 데이터 정합성: 같은 측정 → 무료·유료 데이터 통합 → 컷오프 보정
2. 무료 매력: 가볍고 재밌어야 콘텐츠 유입 클릭·완료율 ↑

### ★주의: 변별력은 유지 (민감도 동일)
- 표현 바꿔도 '변별 강도'는 유지해야 함
- 나쁜 예: "먹는 거 좋아한다😋" → 다들 예 → 변별 안 됨
- 좋은 예: "또래보다 많이 먹는다" → 비교기준 살림 → 변별 O
- ★어린이틱 + 비교기준 유지 = 둘 다 잡기

### 설문 버전 관리
- 같은 문항ID(a1, b1, s1...)에 [유료문항]/[무료문항] 두 표현
- 같은 성격태그(몸/습관/정신)·같은 가중치
- → 한 ID, 두 표현, 같은 측정

### ★무료/유료 차이 = 말투 + 문항 수 (대표님 확정)
```
          무료              유료
말투      친근·가벼움        전문 용어 느낌
문항수    18문항 (3분)       30문항 (정밀)
항목      흡수/활동/저장 (동일)
요소      몸/습관/정신 (동일)
```

- 무료 18문항: 콘텐츠 유입 → 부담 없이 빨리 (완료율↑)
- 유료 30문항: 돈값 → 정밀·변별력·신뢰도↑
- 예:
  · 무료: "밥 먹을 때 '한 그릇 더!' 외친다 🍚"
  · 유료: "또래 대비 1회 섭취량이 많은 편이다"

### ★중요: 문항 수 달라도 0~100 정규화 (필수)
- 무료 18문항 → 흡수력 0~100 환산
- 유료 30문항 → 흡수력 0~100 환산
- ★둘 다 0~100으로 맞춰야 같은 컷오프(40/65) 적용 가능
- → 무료 BioCode와 유료 BioCode가 같은 잣대로 비교됨
- 환산식: (획득점수 / 문항수×최대점) × 100

### 설문 버전 관리 (정리)
- 같은 측정 항목(흡수/활동/저장 × 몸/습관/정신)
- 무료 = 18문항(쉬운 말), 유료 = 30문항(전문 말)
- 무료 18 ⊂ 유료 30 (무료는 유료의 핵심 문항 선별)
- 둘 다 0~100 정규화 → 같은 컷오프
